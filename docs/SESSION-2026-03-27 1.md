### Model: `Play`

```python
class Play(models.Model):
    user_id   = models.IntegerField()
    song      = models.ForeignKey(Song, on_delete=models.CASCADE, related_name='plays')
    played_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        indexes = [
            models.Index(fields=['user_id', '-played_at']),
            models.Index(fields=['song', '-played_at']),
        ]
```

- `user_id` is a plain IntegerField (cross-service reference to auth_user — no FK)
- `song` is a real Django ForeignKey (Song lives in the same core service)
- No unique constraint — the same song can be played many times
- Composite descending indexes for fast "recent plays by user" and "play count
  per song" queries

### Serializers

- `PlaySerializer` — wraps a Play record with a nested `SongSerializer`
  for the `song` field. `user_id` and `played_at` are read-only.

### Views

| Class / Function   | Method | Description |
|--------------------|--------|-------------|
| `RecordPlayView`   | POST   | Accepts `{"song_id": N}`. Validates Song exists, creates Play record. Fire-and-forget from frontend player. |
| `RecentPlaysView`  | GET    | Returns last 10 distinct songs played by current user. Deduplicates using a `seen` set while iterating plays ordered by `-played_at`. |
| `health_check`     | GET    | Standard DB connectivity check. Returns `{"status": "healthy", "service": "history", ...}` |

**Note on `select_related` in `RecentPlaysView`:** Currently uses

## 📦 Part C: `shareapp` — Collaboration Service

### Purpose

Provides read-only access tokens for playlists. A `ShareLink` token allows
another user to view the playlist and listen to songs without being granted
edit access. No `Collaborator` record is created when accessing via ShareLink.

This is distinct from `InviteLink` (collabapp), which grants edit access and
creates a `Collaborator` record on join.

### Key Distinction

| Token Type   | Table                    | On access         | Access level         |
|--------------|--------------------------|-------------------|----------------------|
| `InviteLink` | `collabapp_invitelink`   | Creates Collaborator record | Edit (add/remove/reorder tracks) |
| `ShareLink`  | `shareapp_sharelink`     | Creates nothing   | View-only (listen only) |

### Model: `ShareLink`

```python
class ShareLink(models.Model):
    playlist_id   = models.IntegerField()
    token         = models.UUIDField(default=uuid.uuid4, unique=True, editable=False)
    created_by_id = models.IntegerField()
    is_active     = models.BooleanField(default=True)
    created_at    = models.DateTimeField(auto_now_add=True)
    expires_at    = models.DateTimeField(null=True, blank=True)

    class Meta:
        indexes = [
            models.Index(fields=['token']),
            models.Index(fields=['playlist_id', 'is_active']),
        ]

    @property
    def is_valid(self):
        if not self.is_active:
            return False
        if self.expires_at and timezone.now() > self.expires_at:
            return False
        return True
```

- `playlist_id` and `created_by_id` are plain IntegerFields (cross-service refs)
- `token` is auto-generated UUID, not user-editable
- `is_valid` property checks both `is_active` AND `expires_at` — always use
  this instead of `is_active` alone in view logic
- Schema is identical to `InviteLink` — the behavioural difference is entirely
  in the views (no Collaborator creation on access)

### Serializer

- `ShareLinkSerializer` exposes `is_valid` as a read-only `SerializerMethodField`
- `token`, `created_by_id`, `created_at`, `is_valid` are all read-only

## 🔑 Key Technical Decisions

### 1. historyapp depends on current flat Song model

The `Play` model's FK points to `searchapp.song`. At the time of Commit 1,
`Song` is still the flat model (plain `artist`/`album` CharField fields).
This dependency is correct for now. Commit 2 will rewrite `Song` but will
keep the migration file name as `0001_initial`, so the dependency reference
`('searchapp', '0001_initial')` in historyapp's migration remains valid
without modification.

### 2. shareapp has no Django FK dependencies

`ShareLink` uses `IntegerField` for both `playlist_id` (references core
service's Playlist) and `created_by_id` (references auth service's User).
Since both targets live in different Django applications, real ForeignKey
relationships are not possible at the ORM level. This follows the same
convention used by `collabapp_collaborator` and `collabapp_invitelink`.

### 3. RecentPlaysView deduplication strategy

Rather than a complex SQL DISTINCT ON query (which varies by database backend),
deduplication is done in Python: iterate plays ordered by `-played_at`,
maintain a `seen` set of `song_id` values, append each unseen song to `recent`,
stop at 10. This is simple, portable, and correct. The query is bounded by the
fact that most users have at most a few hundred play records.

### 4. DeactivateShareLinkView uses bulk update

Rather than fetching a single active share link and deactivating it,
`DeactivateShareLinkView` uses `.filter(...).update(is_active=False)` to
deactivate all active share links for the playlist in one query. This handles
the edge case where multiple active share links exist for the same playlist
(which shouldn't happen normally, but could through direct DB manipulation).

### 5. ViewShareLinkView returns playlist_id in response

The collaboration service has no access to Playlist data (that lives in the
core service). `ViewShareLinkView` therefore returns the `playlist_id` integer
from the ShareLink record. The frontend uses this to make a separate call to
`GET /api/playlists/:id/` on the core service to load the playlist contents.

## ⚠️ Known Limitations (To Be Resolved in Later Commits)

1. **historyapp SongSerializer is flat** — Until Commit 2 updates `searchapp`,
   `SongSerializer` returns flat artist/album strings instead of nested objects.
   `RecentPlaysView` will return the correct structure after Commit 2.

2. **No owner enforcement on ShareLink creation** — Any authenticated user can
   create a share link for any `playlist_id`. Owner-only enforcement requires
   a cross-service call to the core service to verify `playlist.owner_id ==
   request.user.id`. This is documented as a known gap to be addressed in
   a future hardening pass.

3. **historyapp Play model meta index limitation** — The `Meta.indexes` with
   descending fields `[..., '-played_at']` creates proper descending indexes
   in PostgreSQL. If the database is ever switched to SQLite (e.g. for unit
   tests), descending index syntax may not be supported. Not a concern for
   this project (PostgreSQL only).
