## 📦 Category A: InviteLink & ShareLink — Expiry Design

### The Design Decision

Previously both `InviteLink.expires_at` and `ShareLink.expires_at` were
`DateTimeField(null=True, blank=True)` — an optional field that could be set
manually. The design is now finalised:

| Aspect | Before | After |
|--------|--------|-------|
| `expires_at` default | None (nullable) | `now() + 30 days` (always set) |
| `expires_at` null | Yes (`null=True`) | No (non-nullable) |
| Manual deactivation | API endpoint existed | No endpoint — time-only expiry |
| `is_valid` logic | `is_active AND (no expires_at OR now <= expires_at)` | `is_active AND now <= expires_at` |

### The `default_expires_at` Function

The same function is defined at module level in both models files:

```python
from datetime import timedelta
from django.utils import timezone

def default_expires_at():
    return timezone.now() + timedelta(days=30)
```

Used as a callable default — Django calls it at instance creation time,
not at class definition time. This ensures each new link gets its own
30-day window starting from the actual creation moment.

### Simplified `is_valid` Property

```python
# Before (required null guard)
@property
def is_valid(self):
    if not self.is_active:
        return False
    if self.expires_at and timezone.now() > self.expires_at:
        return False
    return True

# After (expires_at is always set — no null guard needed)
@property
def is_valid(self):
    return self.is_active and timezone.now() <= self.expires_at
```

## 📦 Category B: Core Service Bug Fixes & Model Corrections

### playlistapp/models.py — Two Field Fixes

**`description` — added `blank=True`**

```python
# Before
description = models.TextField(default='')

# After
description = models.TextField(blank=True, default='')
```

`default=''` alone only sets the DB default. Without `blank=True`, Django REST
Framework's form and serializer validation rejects an empty string as invalid.
Both must be present for description to be genuinely optional.

**`max_songs` — `IntegerField` → `PositiveIntegerField`**

```python
# Before
max_songs = models.IntegerField(default=0)

# After
max_songs = models.PositiveIntegerField(default=0)
```

A negative song limit is semantically meaningless. `PositiveIntegerField` adds
a DB-level check constraint enforcing `max_songs >= 0`.

### trackapp/views.py — Race Condition Fix (transaction.atomic)

The original `TrackListView.post()` ran five separate DB operations
(`exists()`, `count()`, `order_by().first()`, `create()`) without atomicity.
Two concurrent requests could both pass the pre-checks and then collide on the
`(playlist, song)` unique constraint, producing an unhandled 500 error.

**Fix:** wrapped in `transaction.atomic()` with `select_for_update()` on the
playlist row. The `IntegrityError` catch is a belt-and-suspenders safety net.

```python
try:
    with transaction.atomic():
        # Locks the playlist row — concurrent requests queue here
        playlist = Playlist.objects.select_for_update().get(id=playlist_id)

        if Track.objects.filter(playlist=playlist, song=song).exists():
            return Response({'error': 'Song already in playlist'}, ...)
        ...
        track = Track.objects.create(...)
except Playlist.DoesNotExist:
    return Response({'error': 'Playlist not found'}, status=404)
except IntegrityError:
    # Race slipped through — unique constraint caught it
    return Response({'error': 'Song already in playlist'}, status=400)
```

### trackapp/views.py — TrackReorderRemoveView

The view was renamed from `TrackReorderView` to `TrackReorderRemoveView` to
reflect its actual behaviour. The old implementation only updated positions;
if a user removed tracks in the UI and then saved, those tracks would remain
in the DB. The new implementation deletes any tracks missing from the submitted
list before reassigning positions — both operations happen atomically.

```python
def put(self, request, playlist_id):
    ordered_ids = request.data.get('track_ids', [])
    with transaction.atomic():
        # Delete tracks not in the final list (reorder-remove)
        Track.objects.filter(playlist_id=playlist_id).exclude(id__in=ordered_ids).delete()
        # Reassign positions 0, 1, 2, …
        for index, track_id in enumerate(ordered_ids):
            Track.objects.filter(id=track_id, playlist_id=playlist_id).update(position=index)
    return Response({'status': 'reordered'})
```

The URL stays `PUT /<playlist_id>/reorder/` — only the class name changed.

docker-compose exec core uv run python manage.py makemigrations trackapp

## 🔑 Key Technical Decisions

### 1. `expires_at` is non-nullable — no need for null guard

Making `expires_at` always set (via the default callable) eliminates an entire
class of bugs: code that forgot to check `if self.expires_at` before comparing.
The `is_valid` property is now a simple one-liner. Any future code that reads
`expires_at` can trust it is always a real datetime value.

### 2. No manual deactivation endpoint — time is the only invalidation

The decision to remove the deactivation endpoints is a simplification of the
access control model: a link is valid for exactly 30 days, then it expires.
There is no way to revoke it early via the API. This is a deliberate product
decision. The `is_active` field is retained in the model so a Django admin
user could still manually deactivate a link if absolutely necessary — but
there is no API surface for this.

### 3. `default_expires_at` defined at module level, not as a lambda

Django migration serialisation cannot serialise lambda functions or local
functions. Module-level named functions are serialisable and can be referenced
from migration files. Defining `default_expires_at` at the top of the models
file allows the migration to reference it correctly.

### 4. Same function name in both collabapp and shareapp

Both models independently define `default_expires_at()`. They are not shared
because collabapp and shareapp are separate Django apps in the same service
— importing from one into the other would create an unnecessary coupling.
The logic is trivial and the duplication is intentional.
