## 🔧 Changes Applied

### Category A — Collaboration Service

#### `services/collaboration/collabapp/views.py`

**`CollaboratorListView.delete()` — self-removal enforcement**

The previous implementation had only a TODO comment warning about the vulnerability.
Any authenticated user could send `DELETE /collab/<playlist_id>/members/?user_id=<any_id>`
to remove any collaborator.

The quick mitigation compares the `user_id` query param against `request.user.id`
(the JWT-authenticated caller). If they differ, a 403 is returned immediately.

```python
# Before — vulnerable
user_id = request.query_params.get('user_id')
Collaborator.objects.filter(playlist_id=playlist_id, user_id=user_id).delete()
return Response(status=status.HTTP_204_NO_CONTENT)

# After — self-removal only
user_id = request.query_params.get('user_id')
if str(user_id) != str(request.user.id):
    return Response({'error': 'Forbidden'}, status=status.HTTP_403_FORBIDDEN)
Collaborator.objects.filter(playlist_id=playlist_id, user_id=user_id).delete()
return Response(status=status.HTTP_204_NO_CONTENT)
```

**Note:** A full fix (allowing the playlist owner to also remove others) requires a
cross-service HTTP call to the core service to verify `owner_id`. This remains a future
task and is noted in the comment above the implementation.

### Category B — Core Service (trackapp)

#### `services/core/trackapp/views.py`

**`TrackListView.post()` — owner check**

Only the playlist owner should be able to add songs. The check is placed inside the
`transaction.atomic()` block after `select_for_update()` so the ownership read is
consistent with the lock on the playlist row.

```python
playlist = Playlist.objects.select_for_update().get(id=playlist_id)

if playlist.owner_id != request.user.id:
    return Response({'error': 'Forbidden'}, status=status.HTTP_403_FORBIDDEN)
```

**`TrackReorderRemoveView.put()` — payload validation**

The previous implementation used `request.data.get('track_ids', [])`. If `track_ids`
was absent, the default `[]` would cause all tracks to be deleted silently. Three
validations were added before entering the transaction:

1. Key presence check — 400 if `track_ids` key is missing entirely.
2. Type check — 400 if value is not a list.
3. Duplicate check — 400 if the list contains repeated IDs.

Inside the transaction, a membership check verifies every submitted ID belongs to the
target playlist, returning a 400 with the offending ID if not.

```python
if 'track_ids' not in request.data:
    return Response({'error': 'track_ids required'}, status=status.HTTP_400_BAD_REQUEST)
ordered_ids = request.data.get('track_ids')
if not isinstance(ordered_ids, list):
    return Response({'error': 'track_ids must be a list'}, status=status.HTTP_400_BAD_REQUEST)
if len(ordered_ids) != len(set(ordered_ids)):
    return Response({'error': 'track_ids must not contain duplicates'}, status=status.HTTP_400_BAD_REQUEST)
```

**`TrackReorderRemoveView.put()` — playlist lock and 404/403 guards**

The reorder-remove operation previously had no playlist row lock, creating a race
condition with concurrent `TrackListView.post()` calls (both modify positions/tracks
on the same playlist row). The fix mirrors the pattern used in `TrackListView.post()`:

```python
with transaction.atomic():
    playlist = Playlist.objects.select_for_update().get(id=playlist_id)

    if playlist.owner_id != request.user.id:
        return Response({'error': 'Forbidden'}, status=status.HTTP_403_FORBIDDEN)
    ...
except Playlist.DoesNotExist:
    return Response({'error': 'Playlist not found'}, status=status.HTTP_404_NOT_FOUND)
```

### Category C — Model Comment Fix

#### `services/core/trackapp/models.py`

The Meta docstring incorrectly stated that `models.Index(fields=['playlist', 'position'])`
"enforces ordering integrity." A `models.Index` is purely a B-tree performance index. It
does not enforce uniqueness or contiguity. The comment was rewritten:

```python
# Before
# The (playlist, position) index also enforces ordering integrity: after every
# reorder-remove save, positions are reassigned as a contiguous 0-based sequence,
# so this index doubles as a fast ordering lookup.

# After
# The (playlist, position) index is a performance index for fast ordering lookups.
# Contiguous 0-based positions are maintained purely by TrackReorderRemoveView logic,
# not by any database constraint — models.Index does not enforce uniqueness.
```

## 🔍 Security Notes

### Cross-service authorization gap (partial fix)

The collaboration service cannot directly query the core service's `Playlist` table.
The `CollaboratorListView.delete()` fix enforces self-removal only. A future iteration
should implement a lightweight cross-service call (e.g. `GET /core/playlists/<id>/owner/`)
and gate removal on ownership.

### Owner check placement

Both owner checks (`TrackListView.post()` and `TrackReorderRemoveView.put()`) are placed
inside `transaction.atomic()` after `select_for_update()`. This ensures the ownership
read is part of the locked transaction and cannot be invalidated by a concurrent playlist
ownership transfer.
