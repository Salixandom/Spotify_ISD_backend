and verified.

## ­čĆŚ´ŞĆ Search Architecture ÔÇö Hybrid Design

This hybrid approach mirrors Spotify's own search design:

```
GET /api/search/?q=<text>
    Ôćĺ Unified search: returns songs + playlists + artists + albums
    Ôćĺ Frontend renders a "top results" page with all categories
    Ôćĺ Only param: q

GET /api/search/songs/?q=<text>&genre=<g>&sort=<s>&order=<asc|desc>
    Ôćĺ Songs only, with full filtering + sorting
    Ôćĺ Frontend renders the "Songs" tab / filtered view

GET /api/search/playlists/?q=<text>&type=<solo|collaborative>
    Ôćĺ Public playlists only, with optional type filter
    Ôćĺ Frontend renders the "Playlists" tab

GET /api/search/artists/?q=<text>       (pre-existing)
GET /api/search/albums/?q=<text>        (pre-existing)
    Ôćĺ Category-specific, searchable by name
```

The unified endpoint does a single DB hit per category and returns all four in
one response. The category endpoints are used when the user drills into a tab
and may apply additional filters that are meaningless in the global view.

## ­čôŽ Category A: File Changes

### `services/core/searchapp/views.py`

**`SearchView` ÔÇö redesigned as unified search**

Before: searched only songs, returned a flat list.

After: searches all four entity types and returns a grouped response:

```python
return Response({
    'songs':     SongSerializer(songs[:20], many=True).data,
    'playlists': PlaylistSerializer(playlists[:20], many=True).data,
    'artists':   ArtistSerializer(artists[:20], many=True).data,
    'albums':    AlbumSerializer(albums[:20], many=True).data,
})
```

Search logic per type:
- **songs:** `title__icontains | artist__name__icontains | album__name__icontains`
- **playlists:** `visibility='public'` filter + `name__icontains | description__icontains`
- **artists:** `name__icontains`
- **albums:** `name__icontains | artist__name__icontains`

**`SongSearchView` ÔÇö new, at `/api/search/songs/`**

Dedicated song search with full parameter support:

| Param | Type | Description |
|---|---|---|
| `q` | string | Search across title, artist name, album name |
| `genre` | string | Exact genre filter (`iexact`) |
| `sort` | string | One of: `title`, `artist`, `album`, `genre`, `duration`, `year` |
| `order` | string | `asc` (default) or `desc` |

Returns: flat list of `SongSerializer` results (max 20).

**`PlaylistSearchView` ÔÇö new, at `/api/search/playlists/`**

Public-playlist-only search:

| Param | Type | Description |
|---|---|---|
| `q` | string | Search across name and description |
| `type` | string | Optional: `solo` or `collaborative` |

Automatically filters `visibility='public'`. Returns: flat list of
`PlaylistSerializer` results ordered by name (max 20).

## ­čôŽ Category B: SECRET_KEY Fix

### Root Cause

`rest_framework_simplejwt` uses Django's `SECRET_KEY` as the HMAC signing key
for JWT tokens (when no explicit `SIGNING_KEY` is set in `SIMPLE_JWT` config).
All three services must share the same key so tokens issued by `auth` can be
verified by `core` and `collaboration`.

| Service | Before | After |
|---|---|---|
| auth | `change-this-in-production-use-openssl-rand-hex-32` | unchanged |
| collaboration | `change-this-in-production-use-openssl-rand-hex-32` | unchanged |
| core | `supersecretkey123` ÔŁî | `change-this-in-production-use-openssl-rand-hex-32` Ôťů |

### Why `docker compose restart` Was Insufficient

`docker compose restart` sends SIGTERM and restarts the process but does **not**
re-read the `env_file` ÔÇö environment variables are baked in at container
creation time. A full `docker compose down && docker compose up -d` was required
to recreate the container with the updated `.env`.

### File Changed

```
services/core/.env    SECRET_KEY updated to match auth + collaboration
```

## ­čöĹ Key Technical Decisions

### 1. Unified search returns all four types in one request

A single endpoint returning `{songs, playlists, artists, albums}` is more
efficient for the "search results overview" page ÔÇö one HTTP call instead of
four. The frontend can render each section independently and lazily.

### 2. Playlist search filters only `public` playlists

Private playlists are not discoverable via search. A user's own private playlists
are accessible via `/api/playlists/` (which filters by `owner_id`), not via search.

### 3. `SONG_SORT_MAP` kept at module level

`SongSearchView` still uses `SONG_SORT_MAP` for sort field resolution.
It was not removed when `SearchView` was redesigned ÔÇö the unified search
intentionally does not support per-field sorting (that complexity belongs
in the category-specific endpoints).

### 4. `PlaylistSerializer` imported across app boundary

`searchapp/views.py` imports from `playlistapp` ÔÇö both are in the same Django
service (`core`), so this is a clean internal import, not a cross-service call.
