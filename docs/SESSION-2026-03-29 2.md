## ЁЯУЛ Executive Summary

Three features added to `trackapp` and `playlistapp`, all following the same per-user design pattern established during the planning discussion:

**Category A тАФ Solo remove route:** A dedicated `DELETE /<playlist_id>/remove/` endpoint that removes specific tracks by ID without touching positions of the remaining tracks. Previously the only bulk-removal path was `TrackReorderRemoveView` which combined reordering and deletion atomically тАФ useful for saving a drag-and-drop reorder, but unnecessarily heavy for a simple "remove these tracks" action.

**Category B тАФ Per-user playlist archive:** A `UserPlaylistArchive(user_id, playlist)` junction model in `playlistapp`. Users can archive any playlist from their view without deleting it or affecting other collaborators. `GET /api/playlists/` silently excludes archived playlists by default; pass `?include_archived=true` to see them. Design discussion during the session confirmed that a global `is_archived` field on `Playlist` would be wrong for collaborative playlists тАФ one collaborator archiving a shared playlist must not hide it from others.

**Category C тАФ Per-user track hide:** A `UserTrackHide(user_id, track)` junction model in `trackapp`. Users can soft-hide individual tracks within a playlist. `GET /api/tracks/<playlist_id>/` automatically filters out hidden tracks for the requesting user via `.exclude(hidden_by__user_id=request.user.id)`. Matches Spotify's own "Hide song" feature.

Migrations for both new models were run manually by the developer (`0002_userplaylistarchive`, `0002_usertrackhide`).

## ЁЯПЧя╕П Architecture тАФ Per-User Junction Model Pattern

Both `UserPlaylistArchive` and `UserTrackHide` follow the same pattern:

```
user_id (IntegerField)  тЖР  no FK to auth.User тАФ consistent with owner_id pattern in Playlist/Track
playlist/track (FK, CASCADE)
archived_at / hidden_at (DateTimeField, auto_now_add)
unique_together: (user_id, playlist/track)  тЖР  prevents duplicate entries
```

`unique_together` means `get_or_create()` is safe to call repeatedly тАФ archiving an already-archived playlist is a no-op, not an error.

The `user_id` is stored as a plain `IntegerField` (not a FK to `auth.User`) to remain consistent with the rest of the core service, which stores `owner_id` and `added_by_id` the same way тАФ the auth service owns user records and the services communicate via JWT user IDs, not cross-service FKs.

## ЁЯУж Category A: Solo Remove Route

### The Problem with Using `TrackReorderRemoveView` for Simple Removes

`TrackReorderRemoveView` (`PUT /<playlist_id>/reorder/`) accepts a final ordered list and atomically deletes anything not in the list then reassigns positions. It is designed for saving a drag-and-drop UI state. Using it to remove one track requires the client to:

1. Fetch the current track list
2. Remove the target track from the list
3. Send the entire remaining list back

That is three round trips for what should be one. `TrackRemoveView` handles it directly.

### Implementation

```python
class TrackRemoveView(APIView):
    permission_classes = [permissions.IsAuthenticated]

    def delete(self, request, playlist_id):
        track_ids = request.data.get('track_ids', [])
        Track.objects.filter(playlist_id=playlist_id, id__in=track_ids).delete()
        return Response(status=status.HTTP_204_NO_CONTENT)
```

No position reordering тАФ positions of remaining tracks are left as-is. The `TrackReorderRemoveView` should be used when the client also wants to reorder.

**Route:** `DELETE /api/tracks/<playlist_id>/remove/`
**Body:** `{"track_ids": [1, 2, 3]}`
**Response:** `204 No Content`

## ЁЯУж Category B: Per-User Playlist Archive

### New Model тАФ `playlistapp/models.py`

```python
class UserPlaylistArchive(models.Model):
    user_id     = models.IntegerField()
    playlist    = models.ForeignKey(Playlist, on_delete=models.CASCADE, related_name='archived_by')
    archived_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        unique_together = ('user_id', 'playlist')
```

`CASCADE` on playlist delete: if a playlist is deleted, all archive records for it are automatically removed.

### Updated `PlaylistViewSet.get_queryset()`

```python
def get_queryset(self):
    ...
    qs = Playlist.objects.filter(owner_id=self.request.user.id).order_by(order_field)
    if self.request.query_params.get('include_archived') != 'true':
        qs = qs.exclude(archived_by__user_id=self.request.user.id)
    return qs
```

The `.exclude(archived_by__user_id=...)` uses the `related_name='archived_by'` reverse relation on `UserPlaylistArchive`. Django translates this to a `NOT EXISTS` subquery тАФ efficient and correct.

### Archive / Unarchive Endpoints

```
POST   /api/tracks/<playlist_id>/archive/   тЖТ {"status": "archived"}
DELETE /api/tracks/<playlist_id>/archive/   тЖТ 204 No Content
```

`get_or_create` ensures idempotency тАФ archiving twice has no side effect.

## ЁЯФС Key Technical Decisions

### 1. Archive/hide endpoints live in `trackapp`, not `playlistapp`

`playlistapp` owns playlist CRUD. Archive is a separate concern (a user preference, not a mutation of the playlist itself) so it lives in `trackapp` which already serves as the activity layer over playlists. This avoids bloating `playlistapp/views.py` and keeps all user-preference endpoints together.

### 2. `.exclude()` over `.filter(is_hidden=False)`

There is no `is_hidden` field on `Track`. The hide state lives in `UserTrackHide`. Using `.exclude(hidden_by__user_id=...)` is a reverse-relation filter that generates a single efficient `NOT EXISTS` SQL clause тАФ no extra JOIN, no need to touch the Track model itself.

### 3. `get_or_create` for archive/hide тАФ idempotent by design

Calling `POST /archive/` twice does not raise an error. The `unique_together` constraint guarantees at most one record per `(user_id, playlist/track)` pair. This makes the frontend safe to call archive/hide without tracking local state.

### 4. `remove/` does not reorder positions

After a solo remove, remaining track positions are not renumbered. This is intentional тАФ the client has not expressed a reorder intent. If the client also wants to compact positions, it should follow up with a `PUT /reorder/` call.
