**models.py — OVERWRITTEN**

Completely restructured from a flat denormalised table to a true junction table:

| Old fields (removed) | New fields (added) |
|----------------------|---------------------|
| `playlist_id` IntegerField | `playlist` ForeignKey(Playlist, CASCADE) |

Retained: `added_by_id` (IntegerField), `position`, `added_at`.

New constraints:
- `unique_together = ('playlist', 'song')` — same song cannot appear twice in a playlist
- `ordering = ['position']`
- Indexes: `(playlist, position)`, `(playlist, added_at)`, `(added_by_id,)`

**serializers.py — OVERWRITTEN**

```python

    song        = SongSerializer(read_only=True)            # full nested song
    playlist_id = IntegerField(source='playlist.id', read_only=True)
```

Frontend receives the full song object (with nested artist and album) on every track response.

**views.py — OVERWRITTEN**

`TrackListView.post()` — new add-track logic:
1. Resolve `Playlist` from URL `playlist_id` (404 if missing)
2. Read `song_id` from request body (400 if absent)
3. Resolve `Song` (404 if missing)
4. Check `unique_together` — return 400 `"Song already in playlist"` if duplicate
5. Check `max_songs` — return 400 `"playlist_song_limit_reached"` if at limit
6. Auto-assign `position = last.position + 1` (or 0 if empty)
7. Create `Track` and return full serialized response

`TrackListView.get()` — sort support:
```
?sort=custom|title|artist|album|genre|duration|year|added_at
?order=asc|desc
```

All queries use `.select_related('song', 'song__artist', 'song__album')`.

`TrackDetailView.delete()` and `TrackReorderView.put()` — unchanged logic, preserved as-is.

## 🔑 Key Technical Decisions

### 1. trackapp Track.post() resolves Song before creating Track

The frontend sends `{"song_id": 12}`. The view resolves `Song.objects.get(id=song_id)`
before creating the Track record. This is necessary because Track now stores a
real FK to Song — passing a raw integer to the serializer is no longer possible.
The explicit resolution also gives clean 404 error messages.

### 2. max_songs enforcement happens only in add-track view

`Playlist.max_songs = 0` means unlimited. The enforcement code is:
```python
if playlist.max_songs > 0:
    count = Track.objects.filter(playlist=playlist).count()
    if count >= playlist.max_songs:
        return Response({'error': 'playlist_song_limit_reached', ...}, status=400)
```
This check is in `trackapp/views.py` only. The playlist serializer does not
enforce it — the limit applies to tracks, not to playlist creation/update.

### 3. PlaylistSerializer.validate() handles partial updates correctly

```python
def validate(self, data):
    playlist_type = data.get('playlist_type', getattr(self.instance, 'playlist_type', 'solo'))
    visibility    = data.get('visibility',    getattr(self.instance, 'visibility', 'public'))
```

`getattr(self.instance, 'field', default)` reads the current DB value when a
field is omitted from a PATCH request. This ensures the validation works for
full create, full update (PUT), and partial update (PATCH) without false errors.

### 4. JoinView already_member returns HTTP 200, not 409

Per the plan: joining when already a collaborator returns HTTP 200 with
`{"error": "already_member"}`. This is a deliberate idempotency choice —
the client state is valid (user is already a member), it is not an error
in the HTTP sense. The frontend can handle 200 + error key cleanly.

### 5. is_valid always used instead of is_active alone

Both `InviteLink.is_valid` and `ShareLink.is_valid` check two conditions:
`is_active` AND `expires_at`. All view logic in collabapp and shareapp uses
`invite.is_valid` / `share.is_valid`. Direct checks on `is_active` alone
would miss expired links that were never explicitly deactivated.

### 6. historyapp migration unchanged

`historyapp/migrations/0001_initial.py` depends on `('searchapp', '0001_initial')`.
Since the searchapp migration is replaced in-place (same filename `0001_initial`),
the dependency reference resolves correctly without any modification to the
historyapp migration.

## ⚠️ Pre-Deploy Requirement: DB Reset

The searchapp Song table and trackapp Track table schemas are fundamentally
incompatible with their Commit 1 versions. Django cannot auto-generate a
forward migration from the old flat structure to the new FK-based structure.

**Mesbah must run before deploying this commit:**
```bash
docker-compose down -v   # drops all volumes and DB data
docker-compose up -d     # fresh DB, entrypoints auto-run all migrations
```

After the fresh start, migration order within the core service is:
```
searchapp  (Artist → Album → Song)
playlistapp (Playlist — no searchapp dependency, runs independently)
trackapp   (depends on BOTH playlistapp and searchapp)
historyapp (depends on searchapp only)
```
